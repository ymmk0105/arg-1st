# AGENTS.md

## Project
このリポジトリは、Cloudflare Pages で公開する短編ARG（Alternate Reality Game）群と、
その管理用CMSを扱うプロジェクトです。
プレイヤーはブラウザだけで遊べることを前提とし、プレイヤー向け公開物は静的サイトとします。

## Primary goal
最優先は「実際に最初から最後まで遊べること」です。
演出の派手さよりも、導線の明確さ、リンク切れ防止、謎解きの成立を優先してください。

## Technical constraints
- プレイヤー向け公開物は HTML / CSS / JavaScript で配信すること
- Cloudflare Pages で公開できる構成にすること
- プレイヤー向け本編は静的サイトとして成立させること
- CMS / 管理系では Cloudflare Workers / D1 の利用を許可する
- Node.js, npm, Vite, Webpack, TypeScript, Sass などは CMS / 生成基盤で必要なら採用可
- サーバーサイド処理は Cloudflare Workers 上で行うこと
- 外部APIは必要最小限にとどめ、プレイヤー向け本編の進行に必須としないこと
- データベースは Cloudflare D1 を前提とする
- プレイヤー向けページ内リンクは相対パスを基本とすること
- 画像がなくても最低限成立する構成にすること
- localStorage は使用可

## Target experience
- プレイ時間は 20〜40 分程度
- 難易度は初心者〜中級者向け
- ブラウザのみで完結
- スマホでも遊びやすい
- 不穏さはあってよいが、過度なグロ表現や過激表現は避ける
- ホラーに寄せすぎず、「調査」「記録」「違和感」を軸にする

## Content rules
- 実在人物、実在企業、実在事件を使わない
- 個人情報の入力を求めない
- 電話、メール、SNS、外部サイト誘導を前提にしない
- プレイヤーを過度に混乱させる理不尽な謎は避ける
- すべての謎には最低1つ以上のヒント導線を入れる
- 総当たりでしか解けないパスワード設計は禁止
- 謎の答えは作中情報から論理的に導けるようにする

## UX rules
- 1ページの情報量は多すぎないようにする
- 重要情報は視認しやすく配置する
- フォントサイズはスマホでも読みやすくする
- コントラストを確保する
- 正解/不正解時のフィードバックを明確にする
- 入力UIにはプレースホルダや補助文を入れる
- 詰まり対策としてヒントボタンまたは段階ヒントを入れる
- リロードしても進行が極端に失われないよう配慮する

## Game design rules
- 謎は 3〜5 個程度に収める
- 序盤は簡単、中盤で少し考えさせ、終盤で気持ちよく回収する
- 各謎には以下を定義すること
  - 問題
  - ヒント
  - 答え
  - 正解後の導線
- 最終ページでは、真相または回収感が得られるようにする
- 隠し要素を 1 つ入れてもよいが、本筋クリア不能にしないこと

## Visual direction
- 雰囲気は「古い記録サイト」「調査ログ」「保管ファイル」風
- 色数は抑える
- 読みやすさ優先
- アニメーションは最小限
- 重い演出や大きな画像依存は避ける

## File organization
- ファイル名は英小文字とハイフン区切り
- 汎用CSSは `css/style.css`
- 汎用JSは `js/main.js`
- ページ追加時は命名を統一する
- 不要ファイルを増やしすぎない
- README.md にセットアップと公開手順を書く
- D1 migration や CMS 設計資料は用途ごとに整理して配置する

## Required deliverables
最低限、以下を作成すること:
- トップページ
- 導入ページ
- 謎ページ 3〜5 個
- エンディングページ
- README.md
- 謎仕様をまとめたドキュメント（必要なら STORY.md / PUZZLES.md を更新）

## Implementation policy
作業時は次の順で進めること:
1. 全体構成を整理
2. ページ一覧と遷移を決める
3. 謎の仕様を固める
4. HTML/CSS/JS を実装
5. 導線、リンク、入力判定、ヒント動作を確認
6. README.md を整備

## Quality check
完了前に必ず確認すること:
- Cloudflare Pages で壊れる構成になっていないか
- 絶対パスを使っていないか
- リンク切れがないか
- 謎が作中情報だけで解けるか
- ヒント導線があるか
- スマホ表示で崩れないか
- localStorage がなくても最低限遊べるか
- README.md が公開手順として読めるか
- CMS / D1 / 公開フローの設計が現行方針と矛盾していないか

## Output style
- 変更内容は最後に簡潔に要約すること
- どのファイルに何を追加/変更したか明記すること
- 大規模変更の前には意図を短く示すこと
